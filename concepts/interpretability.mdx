# Interpretability in B-cos Networks

B-cos Networks are **inherently interpretable** — they generate explanations as a natural byproduct of their forward pass, without requiring any post-hoc attribution methods. This is achieved through their unique architecture that becomes a dynamic linear model during inference.

## The Core Principle: Dynamic Linear Models

The key insight is that B-cos Networks can be viewed as **dynamic linear models**:

```python
output = W_dynamic @ input
```

Where `W_dynamic` depends on the input but is computed in a way that makes the final operation linear with respect to the input.

### Why Linearity Enables Interpretability

For a linear model `y = Wx`, the contribution of each input feature is simply:
```python
contribution[i] = W[i] * x[i]
```

This means:
- **No ambiguity**: Each input's contribution is exactly its weight times its value
- **Perfect faithfulness**: Contributions sum exactly to the output
- **No approximation**: This is the true computational path

## The `.explain()` Method

Every B-cos model has a built-in `.explain()` method that generates visual explanations:

```python
import torch
from PIL import Image
import matplotlib.pyplot as plt

# Load a pretrained B-cos model
model = torch.hub.load('B-cos/B-cos-v2', 'resnet50', pretrained=True)

# Load and preprocess image
img = model.transform(Image.open('cat.jpg'))
img = img[None].requires_grad_()

# Generate explanation
model.eval()
expl_out = model.explain(img)

# View results
print(f"Predicted class: {expl_out['prediction']}")
plt.imshow(expl_out['explanation'])
plt.show()
```

### What `.explain()` Returns

The `.explain()` method returns a dictionary with:

| Key | Type | Description |
|-----|------|-------------|
| `prediction` | `int` | Predicted class index |
| `explained_class_idx` | `int` | Class being explained (usually same as prediction) |
| `dynamic_linear_weights` | `Tensor` | The gradient `∂y/∂x` (also stored in `input.grad`) |
| `contribution_map` | `Tensor` | Spatial contribution map: `(input * ∂y/∂x).sum(channel_dim)` |
| `explanation` | `np.ndarray` | RGB visualization with alpha channel (ready to display) |

See the implementation in `bcos/common.py:92-190`.

## How It Works: Explanation Mode

The magic happens through **explanation mode**, which temporarily modifies the computation graph:

### Standard Forward Pass (Training)

```python
# Full B-cos computation
linear_out = input @ ŵ.T
dynamic_scale = |cos(input, ŵ)|^(B-1)
output = dynamic_scale * linear_out  # Nonlinear!
```

Here, the output depends **nonlinearly** on the input because `dynamic_scale` is computed from `input`.

### Explanation Mode (Inference)

```python
with model.explanation_mode():
    # Dynamic weights are DETACHED from input
    linear_out = input @ ŵ.T
    dynamic_scale = |cos(input, ŵ)|^(B-1)
    dynamic_scale = dynamic_scale.detach()  # <-- Key step!
    output = dynamic_scale * linear_out  # Now LINEAR w.r.t input!
```

By detaching the dynamic scale, the computation becomes:
```python
output = (constant) * (input @ ŵ.T)
```

This is **linear** in `input`, so gradients are meaningful!

### The DetachableModule

Explanation mode is implemented via `DetachableModule` (see `bcos/modules/common.py:8-34`):

```python
class DetachableModule(nn.Module):
    def __init__(self):
        super().__init__()
        self.detach = False  # Normal mode
    
    def set_explanation_mode(self, activate: bool = True):
        self.detach = activate
```

When `self.detach = True`, B-cos modules detach their dynamic weights:

```python
# From BcosConv2d forward pass
maybe_detached_out = out
if self.detach:
    maybe_detached_out = out.detach()
    norm = norm.detach()

if self.b == 2:
    dynamic_scaling = maybe_detached_out.abs() / norm
else:
    abs_cos = (maybe_detached_out / norm).abs() + 1e-6
    dynamic_scaling = abs_cos.pow(self.b - 1)

out = dynamic_scaling * out  # Linear if self.detach=True
```

See `bcos/modules/bcosconv2d.py:166-178` for the full implementation.

## Gradient-Based Explanations

In explanation mode, the gradient `∂output/∂input` has a special interpretation:

```python
with model.explanation_mode():
    output = model(input)
    output[class_idx].backward()
    
    # input.grad is the dynamic linear weight!
    explanation_weights = input.grad
```

### What Are These Weights?

The gradient represents the **effective linear transformation** from input to output:

```python
W_dynamic[layer_L, layer_1] = ∏(layer=1 to L) |cos(x_layer, ŵ_layer)|^(B-1) · ŵ_layer
```

These are the **weights that the network actually uses** for this specific input.

### Why This Is Better Than Post-Hoc Methods

Traditional attribution methods like:
- **GradCAM**: Uses gradients at intermediate layers (incomplete path)
- **Integrated Gradients**: Approximates by integrating along a path
- **LRP**: Requires custom propagation rules

B-cos explanations:
- ✅ **Exact**: Not an approximation
- ✅ **Faithful**: Sum of contributions equals output
- ✅ **Efficient**: Single backward pass
- ✅ **No hyperparameters**: No baseline or path to choose

## Visualization: From Gradients to Images

The `.explain()` method converts gradients into intuitive visualizations using `gradient_to_image()`:

### Step 1: Compute Contribution Map

```python
# Element-wise product and sum over channels
contribution_map = (input * input.grad).sum(dim=1)  # [H, W]
```

This shows **how much each spatial location contributed** to the prediction.

### Step 2: Generate Color Explanation

For RGB inputs (converted to 6 channels with inverse), the color explanation shows **which color channels mattered**:

```python
def gradient_to_image(image, linear_mapping, smooth=15, alpha_percentile=99.5):
    # Contribution per location
    contribs = (image * linear_mapping).sum(0, keepdim=True)
    
    # Normalize gradients to get direction
    rgb_grad = linear_mapping / (linear_mapping.abs().max(0, keepdim=True).values + 1e-12)
    rgb_grad = rgb_grad.clamp(min=0)  # Only positive weights
    
    # Normalize (R, 1-R), (G, 1-G), (B, 1-B) pairs
    rgb_grad = rgb_grad[:3] / (rgb_grad[:3] + rgb_grad[3:] + 1e-12)
    
    # Alpha = strength of gradient
    alpha = linear_mapping.norm(p=2, dim=0, keepdim=True)
    alpha = torch.where(contribs < 0, 1e-12, alpha)  # Only positive contributions
    
    # Smooth and clip alpha
    if smooth:
        alpha = F.avg_pool2d(alpha, smooth, stride=1, padding=(smooth - 1) // 2)
    alpha = (alpha / torch.quantile(alpha, q=alpha_percentile / 100)).clip(0, 1)
    
    return torch.cat([rgb_grad, alpha], dim=0).permute(1, 2, 0)  # [H, W, 4]
```

See `bcos/common.py:389-435` for the complete implementation.

### Interpretation

The resulting RGBA image shows:
- **RGB values**: Which color direction the network attended to
- **Alpha channel**: How strongly each location contributed
- **Transparency**: Low-contributing regions fade out

## Contribution Maps

For a more quantitative view, use `.plot_contribution_map()`:

```python
expl_out = model.explain(img)
model.plot_contribution_map(expl_out['contribution_map'])
plt.colorbar()
plt.show()
```

This shows:
- **Red regions**: Positive contribution (supports the prediction)
- **Blue regions**: Negative contribution (contradicts the prediction)
- **White regions**: Neutral (no contribution)

The contributions are:
```python
contribution[h, w] = Σ_c (input[c, h, w] * grad[c, h, w])
```

This **exactly sums to the output logit** (up to numerical precision).

## Advanced: The `attribute()` Method

For compatibility with the Captum library, B-cos models also provide an `.attribute()` method:

```python
# Requires: pip install captum
from captum.attr import IntegratedGradients

# Use B-cos's built-in attribution (faster and exact)
attribution = model.attribute(image, target=class_idx)

# This uses IxG with detached dynamic weights
# See bcos/common.py:282-319
```

This method:
1. Activates explanation mode
2. Uses Input × Gradient attribution from Captum
3. Returns the same contributions as `.explain()` but in Captum's format

## Practical Example: Understanding a Prediction

Let's walk through a complete example:

```python
import torch
from PIL import Image
import matplotlib.pyplot as plt

# Load model and image
model = torch.hub.load('B-cos/B-cos-v2', 'resnet50', pretrained=True)
img_pil = Image.open('dog.jpg')
img = model.transform(img_pil).unsqueeze(0).requires_grad_()

# Explain
model.eval()
with torch.no_grad():  # For efficiency in the outer scope
    expl = model.explain(img)

print(f"Prediction: Class {expl['prediction']}")
print(f"Contribution map sum: {expl['contribution_map'].sum().item():.4f}")
print(f"Output logit: {expl['prediction']}")

# Visualize
fig, axes = plt.subplots(1, 3, figsize=(15, 5))

axes[0].imshow(img_pil)
axes[0].set_title('Original Image')
axes[0].axis('off')

axes[1].imshow(expl['explanation'])
axes[1].set_title('B-cos Explanation')
axes[1].axis('off')

model.plot_contribution_map(expl['contribution_map'], ax=axes[2])
axes[2].set_title('Contribution Map')

plt.tight_layout()
plt.show()
```

## Limitations and Considerations

### When Explanation Mode Matters

⚠️ **During Training**: Do NOT use explanation mode
```python
# WRONG - breaks training!
with model.explanation_mode():
    loss = loss_fn(model(x), y)
    loss.backward()
```

Explanation mode should only be used during inference for generating explanations.

### Computational Cost

Generating explanations requires:
1. Forward pass (normal)
2. Backward pass to compute gradients
3. Visualization processing

This is still more efficient than iterative methods like Integrated Gradients.

### Interpretation Caveats

1. **Explanations show correlations, not causation**: High contribution doesn't mean "causes" the prediction
2. **Network quality matters**: Interpretability doesn't guarantee correctness
3. **Smooth parameter affects visualization**: Higher smoothing makes explanations more visually coherent but less spatially precise

## Comparison with Other Methods

| Method | Type | Faithful | Efficient | Setup Required |
|--------|------|----------|-----------|----------------|
| B-cos | Inherent | ✅ Exact | ✅ 1 pass | None |
| GradCAM | Post-hoc | ⚠️ Approximate | ✅ 1 pass | Choose layer |
| Integrated Gradients | Post-hoc | ✅ Exact (with infinite steps) | ❌ Many passes | Choose baseline |
| SHAP | Post-hoc | ✅ Exact | ❌ Many passes | Choose background |
| LRP | Post-hoc | ⚠️ Approximate | ✅ 1 pass | Define rules |
